Voici le tutoriel complet en lignes de commande Bash (à exécuter en tant que `root` sur votre serveur Debian) pour reproduire l'environnement RDP MFA.

## 1. Installation des dépendances système

On commence par installer tous les prérequis nécessaires pour la compilation des modules Python, la base de données MariaDB, et le serveur web Apache avec son module WSGI.

```bash
apt update && apt upgrade -y
apt install -y python3 python3-venv python3-dev \
    mariadb-server mariadb-client libmariadb-dev libmariadb-dev-compat \
    apache2 libapache2-mod-wsgi-py3 ssl-cert \
    gcc pkg-config libffi-dev libssl-dev libjpeg-dev zlib1g-dev
```

## 2. Préparation de la base de données

privacyIDEA a besoin d'une base de données SQL pour stocker sa configuration et les jetons. Nous créons une base MariaDB dédiée.

```bash
mysql -u root -e "CREATE DATABASE pi CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;"
mysql -u root -e "GRANT ALL PRIVILEGES ON pi.* TO 'pi'@'localhost' IDENTIFIED BY 'SecretDBPass123!';"
mysql -u root -e "FLUSH PRIVILEGES;"
```

*(N'hésitez pas à modifier `SecretDBPass123!` par un mot de passe fort).*

## 3. Création de l'environnement virtuel Python

L'installation de l'application se fait de manière isolée dans le répertoire `/opt/privacyidea`.

```bash
mkdir -p /opt/privacyidea
python3 -m venv /opt/privacyidea
source /opt/privacyidea/bin/activate

# Mise à jour des outils de base de l'environnement
pip install -U pip setuptools wheel

# Installation du connecteur MySQL et de privacyIDEA
pip install mysqlclient
pip install privacyidea
```

## 4. Configuration de privacyIDEA

Les fichiers de configuration et les journaux doivent être placés dans des répertoires dédiés. Nous allons générer automatiquement les clés de chiffrement de l'application via OpenSSL.

```bash
mkdir -p /etc/privacyidea /var/log/privacyidea

# Création du fichier de configuration principal
cat <<EOF > /etc/privacyidea/pi.cfg
import logging
SQLALCHEMY_DATABASE_URI = 'mysql://pi:SecretDBPass123!@localhost/pi'
SECRET_KEY = '$(openssl rand -hex 24)'
PI_PEPPER = '$(openssl rand -hex 24)'
PI_ENCFILE = '/etc/privacyidea/enckey'
PI_AUDIT_MODULE = 'privacyidea.lib.auditmodules.sqlaudit'
PI_AUDIT_SQL_URI = SQLALCHEMY_DATABASE_URI
PI_LOGFILE = '/var/log/privacyidea/privacyidea.log'
PI_AUDIT_KEY_PRIVATE = '/etc/privacyidea/private.pem'
PI_AUDIT_KEY_PUBLIC = '/etc/privacyidea/public.pem'
EOF

# Génération de la clé de chiffrement du coffre-fort
pi-manage create_enckey

# Création des tables dans la base de données MariaDB
pi-manage create_tables

# Création du compte administrateur local (l'invite vous demandera de définir un mot de passe)
pi-manage admin add admin -e admin@localhost
```

Nous allons utiliser OpenSSL pour générer directement une clé privée et une clé publique au bon format dans le dossier `/etc/privacyidea/` :

```bash
openssl genrsa -out /etc/privacyidea/private.pem 2048
openssl rsa -in /etc/privacyidea/private.pem -pubout -out /etc/privacyidea/public.pem
```

Il est impératif que l'utilisateur Apache (`www-data`) puisse lire ces clés pour signer les logs :

```bash
chown www-data:www-data /etc/privacyidea/private.pem /etc/privacyidea/public.pem
chmod 600 /etc/privacyidea/private.pem
chmod 644 /etc/privacyidea/public.pem
```

## 5. Configuration d'Apache2 (WSGI)

Pour exposer l'application, nous allons lier l'environnement virtuel à Apache2 en HTTPS.

**Créer le fichier de lancement WSGI :**

```bash
cat <<EOF > /etc/privacyidea/privacyideaapp.wsgi
import sys
sys.stdout = sys.stderr
from privacyidea.app import create_app
application = create_app(config_name="production", config_file="/etc/privacyidea/pi.cfg")
EOF
```

**Attribuer les bons droits à l'utilisateur du serveur web :**

```bash
chown -R www-data:www-data /etc/privacyidea /var/log/privacyidea
chmod 600 /etc/privacyidea/pi.cfg
chmod 600 /etc/privacyidea/enckey
```

**Créer le VirtualHost Apache :**

```bash
cat <<EOF > /etc/apache2/sites-available/privacyidea.conf
<VirtualHost _default_:443>
    ServerAdmin webmaster@localhost
    DocumentRoot /var/www/html

    # Utilisation du certificat auto-signé généré par Debian (ssl-cert)
    SSLEngine on
    SSLCertificateFile /etc/ssl/certs/ssl-cert-snakeoil.pem
    SSLCertificateKeyFile /etc/ssl/private/ssl-cert-snakeoil.key

    # Configuration WSGI pointant vers l'environnement virtuel
    WSGIDaemonProcess privacyidea user=www-data group=www-data processes=1 threads=15 display-name=%{GROUP} python-home=/opt/privacyidea
    WSGIProcessGroup privacyidea
    WSGIPassAuthorization On
	WSGIApplicationGroup %{GLOBAL}
    WSGIScriptAlias / /etc/privacyidea/privacyideaapp.wsgi

    <Directory /etc/privacyidea>
        Require all granted
    </Directory>
</VirtualHost>
EOF
```

## 6. Activation et Démarrage

Enfin, désactivez le site par défaut, activez les modules requis et redémarrez Apache pour appliquer la configuration.

```bash
a2enmod wsgi ssl rewrite
a2ensite privacyidea
a2dissite 000-default
systemctl restart apache2
```

Votre serveur privacyIDEA est désormais opérationnel sur `https://<IP_DE_VOTRE_DEBIAN>`. Vous pouvez vous y connecter avec le compte `admin` créé à l'étape 4

## Connexion RDP

Par conception, l'écran de connexion de Windows (LogonUI) affiche **tous** les fournisseurs d'identifiants (Credential Providers) installés sur le système. L'installation de l'agent privacyIDEA ajoute une nouvelle "tuile" pour l'authentification MFA, mais ne désactive pas par défaut la tuile standard "Mot de passe" de Microsoft. L'utilisateur peut donc simplement choisir l'option classique et contourner le MFA.\[^11\_1]

Pour forcer le MFA, vous devez activer le composant de **Filtre** de privacyIDEA, dont le rôle est de masquer les autres fournisseurs d'identifiants.

En tant qu'administrateur système, vous avez deux méthodes pour corriger cela :

### Méthode 1 : Via l'installeur (Le plus simple)

Relancez l'installeur `.msi` du client privacyIDEA sur la machine Windows. Lors des étapes de configuration, cochez la case indiquant de définir privacyIDEA comme **fournisseur par défaut / cacher les autres fournisseurs**.

### Méthode 2 : Via le Registre (Powershell)

Vous pouvez modifier le comportement à la volée en ajoutant une clé de registre spécifique. L'agent privacyIDEA lit sa configuration dans `HKLM\SOFTWARE\NetKnights GmbH\PrivacyIDEA-CP`.\[^11\_2]

Ouvrez une console PowerShell en tant qu'administrateur sur votre machine cible et lancez :

```powershell
New-ItemProperty -Path "HKLM:\SOFTWARE\NetKnights GmbH\PrivacyIDEA-CP" -Name "enable_filter" -Value "1" -PropertyType String -Force
```

*(Si la valeur `1` est présente, le filtre s'active au prochain verrouillage/redémarrage et l'utilisateur n'aura plus d'autre choix que d'utiliser la tuile privacyIDEA).*

### ⚠️ Avertissement d'Administration

Si vous activez ce filtre en laboratoire et que votre serveur Debian (privacyIDEA) est éteint ou inaccessible, **vous serez totalement bloqué** à l'extérieur de votre machine Windows car le seul moyen de connexion exigera le MFA.

Pour éviter cela, il est fortement recommandé de créer une exception pour le compte Administrateur local du serveur (qui pourra toujours se connecter sans MFA en cas de crise). Toujours dans PowerShell :

```powershell
$piCpKey = "HKLM:\SOFTWARE\NetKnights GmbH\PrivacyIDEA-CP"

# Exclusion d'un compte
New-ItemProperty -Path $piCpKey -Name "excluded_account" -Value "MONSERVEUR\Administrateur" -PropertyType String -Force

# Exclusion d'un groupe
New-ItemProperty -Path $piCpKey -Name "excluded_group" -Value "CONTOSO\GG-PrivacyIDEA-Bypass" -PropertyType String -Force
```

Fermez la session RDP et reconnectez-vous, vous ne devriez plus avoir le choix d'utiliser l'authentification simple.

# Distribution des QR codes par le portail

Le moyen le plus simple et le plus natif prévu par privacyIDEA pour distribuer les QR codes (tokens TOTP) aux utilisateurs finaux est le **Portail Libre-Service** (User Self-Service Portal).

Il n'est en effet pas recommandé que l'administrateur génère lui-même les QR codes pour les envoyer par e-mail, car cela pose d'évidents problèmes de sécurité (le secret circule en clair sur le réseau et l'administrateur y a eu accès).

Voici comment fonctionne ce flux automatisé et sécurisé pour vos utilisateurs :

## 1. Le portail Libre-Service (Self-Service)

L'interface web de privacyIDEA (celle sur laquelle vous vous connectez actuellement en tant qu'`admin`) est en réalité "double".
Si un utilisateur standard du domaine Active Directory s'y connecte avec ses identifiants AD habituels (sans droits d'administration), il accédera à une version bridée de l'interface appelée le *Self-Service Portal*.\[^12\_2]

## 2. L'enrôlement (Enrollment) par l'utilisateur

Lorsqu'un utilisateur se connecte pour la première fois à ce portail avec son mot de passe Windows :

1. Il arrive sur une page lui proposant de "S'enrôler" (Enroll a token).
2. Il choisit "TOTP".
3. Le serveur génère le QR code directement sur son écran.
4. L'utilisateur le scanne avec son application (Microsoft Authenticator, FreeOTP, Google Authenticator ou l'application officielle privacyIDEA).\[^12\_1]

## 3. Comment configurer ce comportement via les Polices

Pour automatiser et sécuriser ce processus, vous devez créer des **Policies** (Règles) dans l'interface d'administration de privacyIDEA :

### 1. Politique de connexion (Scope : Authentication)

Par défaut, privacyIDEA s'attend à ce qu'on se connecte avec un token (MFA). Mais un nouvel utilisateur n'a pas encore de token ! Il faut donc lui donner le droit de se connecter **uniquement avec son mot de passe Active Directory** tant qu'il n'a pas de token.

- **Nom de la policy :** `Connexion_Nouveaux_Users` (par exemple)
- **Scope :** `authentication`
- **Action :** Cherchez la ligne `passthru` et sélectionnez `userstore`.
- *(Optionnel mais recommandé pour la sécurité)* Dans la partie "Conditions", vous pouvez spécifier que cette règle ne s'applique qu'au client correspondant à l'adresse IP de votre serveur web, pour éviter que le "passthru" (sans MFA) ne s'applique aussi au RDP.
- Sauvegardez.

### 2. Politique de l'interface Web (Scope : User)

Cette règle définit ce que l'utilisateur a le droit de voir et de faire lorsqu'il arrive sur le portail web (Self-Service).

- **Nom de la policy :** `Droits_Enrollment_User`
- **Scope :** `user`
- **Action :** Cherchez la catégorie `enroll` (ou tapez dans la barre de recherche de l'action) et cochez **`enrollTOTP`**. Vous pouvez aussi y cocher `delete` (pour qu'il puisse supprimer son propre token en cas de besoin)
- **Action :** Cherchez l'action **`totp_hashlib`** Dans le champ de texte associé, tapez `sha1` (ou `sha256` si vos applications le supportent). Effet : Le menu déroulant de l'algorithme sera supprimé de l'interface et la valeur saisie sera forcée
- **Action :** Cherchez l'action **`totp_otplen`** Dans le champ de texte, tapez `6`. Effet : L'option permettant de choisir entre 6 et 8 chiffres disparaîtra.
- **Action :** Cherchez l'action **`totp_timestep`** Dans le champ de texte, tapez `30`. Effet : La durée de validité du code sera fixée à 30 secondes et l'option disparaîtra.
- **Action :** Cherchez et cochez l'action **`totp_force_server_generate`** Effet : Cela empêche l'utilisateur d'importer sa propre clé (Seed) manuellement.
- Optionnel : Par défaut, l'utilisateur peut donner un nom à son jeton (ex: "Mon iPhone"). Si vous souhaitez également masquer ce champ pour que l'enrôlement se fasse littéralement en un seul clic sans aucune saisie : Assurez-vous que l'action **`setdescription`** n'est **PAS cochée**. Si elle n'est pas autorisée par la politique, le champ de description est automatiquement masqué.
- Sauvegardez.

### 3. Politique de l'interface et de l'Assistant (Scope : WebUI)

C'est ici qu'on force l'apparition de l'assistant d'enrôlement au centre de l'écran.

- **Nom de la policy :** `Assistant_WebUI`
- **Scope :** `webui`
- **Action :**
  - Cochez **`tokenwizard`** (cela active l'assistant automatique pour les utilisateurs sans token).
  - *(Optionnel)* Spécifiez **`default_tokentype`** avec la valeur `totp` (pour que l'assistant propose directement le QR code sans demander à l'utilisateur de choisir le type de jeton).
- Sauvegardez.

### Étape supplémentaire recommandée : Vérification (Scope : Enrollment)

Pour être certain que l'utilisateur a bien scanné le QR code avant de valider, il est conseillé d'ajouter une 4ème politique :

- **Scope :** `enrollment`
- **Action :** Cherchez **`verify_enrollment`** et tapez `TOTP` dans le champ.
  Cela obligera l'utilisateur à taper le premier code à 6 chiffres affiché sur son téléphone pour prouver que l'enrôlement a bien fonctionné avant que le token ne soit activé.
- Sauvegardez.

### Résultat pour vos utilisateurs :

Dès que ces 3 règles sont actives :

1. L'utilisateur (Jean) va sur `https://votre-serveur-privacyidea/`.
2. Il tape `Jean` et son mot de passe Windows habituel (Policy 1 : *Passthru* l'autorise).
3. L'interface charge (Policy 2 : Il a les bons droits *WebUI*).
4. privacyIDEA détecte que Jean n'a aucun token assigné à son compte. L'assistant (Policy 3 : *Enrollment wizard*) se déclenche instantanément : une fenêtre apparaît au milieu de l'écran avec le QR Code généré.
5. Jean le scanne avec son smartphone, entre le premier code à 6 chiffres pour valider, et son token est définitivement lié à son compte AD.

Lors de ses prochaines connexions RDP, le composant Windows exigera alors son code TOTP, et tout fonctionnera automatiquement de bout en bout.

# Auto-enrôlement par Email

L'écran de verrouillage de Microsoft (LogonUI) est extrêmement restrictif par mesure de sécurité : il ne permet pas aux fournisseurs d'identifiants (Credential Providers) d'injecter des images dynamiques (comme un QR Code). Si vous tenez absolument à utiliser des applications comme Google ou Microsoft Authenticator (TOTP), vos utilisateurs devront passer une première fois par le portail web avant d'ouvrir leur session RDP.

Cependant, il existe une alternative très populaire dans privacyIDEA pour rendre le processus 100% transparent lors d'une première connexion RDP, sans passer par le web : l'Auto-enrôlement par Email.

L'alternative : L'Auto-enrôlement par Email au premier login
Au lieu d'utiliser un QR code, vous pouvez configurer privacyIDEA pour qu'il réagisse intelligemment à la première connexion d'un utilisateur sur un serveur RDS.

Voici comment fonctionne ce flux pour l'utilisateur :

1. L'utilisateur lance sa connexion RDP et tape son Login et son Mot de passe habituels.

2. privacyIDEA reçoit la demande, interroge la base et constate que l'utilisateur n'a aucun token.

3. Au lieu de rejeter la connexion, le serveur va lire l'attribut mail de l'utilisateur dans l'Active Directory.

4. Il génère automatiquement un "Token Email" en arrière-plan, et envoie instantanément un code à 6 chiffres sur la boîte mail du collaborateur.
   ​

5. L'écran RDP Windows affiche alors le second champ en demandant le code. L'utilisateur regarde ses mails sur son téléphone, tape le code, et la session s'ouvre !

## Logique globale

La condition native `user_token_number` dans les Event Handlers compte **tous** les tokens de l'utilisateur, sans filtrage par type . La stratégie retenue consiste donc à rendre le token email **éphémère** : créé juste avant l'authentification, supprimé juste après le succès. Ainsi, les utilisateurs se retrouvent toujours dans l'un de ces deux états stables :

- **0 token** → pas encore de TOTP → auto-enrôlement du token email
- **≥ 1 token(s) TOTP** → pas de création de token email, défi TOTP direct

## Policy d'authentification

Créer (ou modifier) la policy dans *Config → Policies → New Policy* :

| Champ      | Valeur                     |
| :--------- | :------------------------- |
| **Nom**    | `pol_auth_rds`             |
| **Scope**  | Authentication             |
| **Realm**  | ton realm AD               |
| **Action** | `otppin = userstore`       |
| **Action** | challenge\_response = totp |

L'action `otppin = userstore` est indispensable : le mot de passe AD est utilisé comme PIN du token, ce qui permet au Credential Provider RDS de déclencher le challenge sans mot de passe OTP séparé.

***

## Event Handler 1 — Création automatique du token email (PRE)

C'est le handler documenté comme exemple canonique par l'équipe privacyIDEA.

*Config → Event Handlers → Add Event Handler*

| Champ              | Valeur                |
| :----------------- | :-------------------- |
| **Nom**            | `EH_AutoEnroll_Email` |
| **Event**          | `validate/check`      |
| **Position**       | `pre`                 |
| **Handler Module** | Token Handler         |
| **Action**         | `enroll`              |

**Conditions :**

| Section | Clé                 | Valeur |
| :------ | :------------------ | :----- |
| User    | `user_token_number` | `0`    |

**Options de l'action `enroll` :**

| Clé             | Valeur       |
| :-------------- | :----------- |
| `tokentype`     | `email`      |
| `dynamic_email` | `True`       |
| `user`          | `True`       |
| `realm`         | ton realm AD |

Avec `dynamic_email = True`, l'adresse est lue dynamiquement depuis le userstore (AD) à chaque authentification — elle n'est jamais stockée dans le token. Si tu souhaites utiliser un serveur SMTP spécifique plutôt que le serveur système par défaut, ajoute `smtp_identifier = <nom_de_ta_config_SMTP>`.

***

## Event Handler 2 — Suppression du token email après succès (POST)

*Config → Event Handlers → Add Event Handler*

| Champ              | Valeur                       |
| :----------------- | :--------------------------- |
| **Nom**            | `EH_Delete_Email_On_Success` |
| **Event**          | `validate/check`             |
| **Position**       | `post`                       |
| **Handler Module** | Token Handler                |
| **Action**         | `delete`                     |

**Conditions :**

| Section | Clé            | Valeur  |
| :------ | :------------- | :------ |
| Result  | `result_value` | `True`  |
| Token   | `tokentype`    | `email` |

Le handler `delete` en position `post` supprime le token identifié dans l'événement (le token email qui vient d'être utilisé avec succès). Cela ramène `user_token_number` à `0`, de sorte que la prochaine connexion sans TOTP recréera un token email éphémère.\[^1\_1]

***

## Nettoyage des tokens email orphelins (recommandé)

Si un utilisateur commence l'authentification (le token email est créé) mais ne finalise pas la saisie OTP (fenêtre fermée, mauvais mot de passe AD, etc.), le token email reste orphelin et bloque la logique au prochain essai. La solution recommandée est un **token janitor** en cron :\[^1\_1]

```bash
# /etc/cron.d/privacyidea-email-cleanup
# Supprime les tokens email non utilisés depuis plus de 15 minutes
*/5 * * * * privacyidea /opt/privacyidea/bin/privacyidea-token-janitor \
  find \
  --tokentype email \
  --last-auth-before "$(date +'%Y-%m-%d %H:%M:%S' --date 'now - 15 minutes')" \
  --action delete >> /var/log/privacyidea-janitor.log 2>&1
```

## Clés de registre du Credential Provider (Serveur RDS)

Chemin : `HKLM\SOFTWARE\NetKnights GmbH\PrivacyIDEA-CP\`

### Connexion au serveur

| Clé (REG\_SZ)           | Rôle                                     |
| :---------------------- | :--------------------------------------- |
| `hostname`              | URL complète ou IP du serveurprivacyIDEA |
| `custom_port`           | Port personnalisé (défaut : 443)         |
| `ssl_ignore_invalid_cn` | `1` pour ignorer les erreurs CN SSL      |
| `ssl_ignore_unknown_ca` | `1` pour ignorer les CA inconnues        |

### Gestion du realm (ce qui remplace mon `realm` erroné)

| Clé             | Type          | Rôle                                                                 |
| :-------------- | :------------ | :------------------------------------------------------------------- |
| `default_realm` | REG\_SZ       | Realm envoyé si aucun mapping ne correspond                          |
| `send_upn`      | REG\_SZ (`1`) | Envoie l'UPN complet (`user@domain.local`) plutôt que le login court |

Le mapping domaine → realm se fait dans la **sous-clé dédiée** :\[^3\_1]

```
HKLM\SOFTWARE\NetKnights GmbH\PrivacyIDEA-CP\realm-mapping\
```

Chaque entrée REG\_SZ = `Nom : domaine_windows` → `Valeur : nom_realm_privacyIDEA`

### Comportement de login (clés utiles pour RDS)

| Clé (REG\_SZ)                   | Rôle                                                                    |
| :------------------------------ | :---------------------------------------------------------------------- |
| `cpus_logon`                    | Comportement lors du logon (ex: `1e` = remote uniquement + CP exclusif) |
| `otp_fail_return_to_first_step` | `1` = retour à l'écran password+OTP si l'OTP échoue                     |
| `no_default`                    | `1` = CP non sélectionné par défaut                                     |
| `log_sensitive`                 | `1` = log des données sensibles pour debug (à supprimer après)          |

# Informations utiles

## Décodage de `cpus_logon` du Credential Provider

Chemin : `HKLM\SOFTWARE\NetKnights GmbH\PrivacyIDEA-CP\`

La valeur est composée d'**un chiffre** (le périmètre) + **une lettre** (le comportement) :\[^4\_1]

### Le chiffre — périmètre d'application

| Valeur | Signification                   |
| :----- | :------------------------------ |
| `0`    | Remote (RDP) **et** local       |
| `1`    | Remote (RDP) **uniquement**     |
| `2`    | Local **uniquement**            |
| `3`    | Désactivé (ni remote, ni local) |

### La lettre — mode de cohabitation des CP

| Valeur | Signification                                                                                 |
| :----- | :-------------------------------------------------------------------------------------------- |
| `e`    | **Exclusif** — seul le CP privacyIDEA est disponible, tous les autres CP Windows sont masqués |
| `d`    | **Additionnel** — le CP privacyIDEA s'ajoute aux CP Windows existants                         |

## Délai de déconnexion de l'interface WebUI

Dans privacyIDEA, le délai de déconnexion de l'interface WebUI est contrôlé via le système de **politiques (policies)**, ce qui permet effectivement de le cibler sur un seul compte admin.

### Paramètre concerné : `logout_time`

La politique `logout_time` (scope **WebUI**) définit le délai d'inactivité avant déconnexion. Par défaut, il est fixé à **120 secondes**. L'action complémentaire `timeout_action` détermine le comportement : `lockscreen` (verrouillage, par défaut) ou `logout` (déconnexion complète).

### Créer une politique ciblée pour un seul admin

Puisque les policies peuvent être filtrées par **realm** et par **utilisateur**, il est tout à fait possible de n'appliquer la règle qu'à un compte spécifique :

1. Aller dans **Config → Policies → New Policy**

2. Renseigner les champs suivants :
   | Champ      | Valeur                                     |
   | :--------- | :----------------------------------------- |
   | **Name**   | `admin_long_session` (ou autre)            |
   | **Scope**  | `WebUI`                                    |
   | **Realm**  | Le realm admin concerné (ex. `adminrealm`) |
   | **User**   | Le nom exact du compte admin ciblé         |
   | **Action** | `logout_time = 3600` (en secondes, ici 1h) |

3. Sauvegarder la politique.

### Points importants

- La valeur de `logout_time` est en **secondes** : `3600` = 1h, `7200` = 2h, etc.
- Si plusieurs policies `logout_time` s'appliquent au même utilisateur, privacyIDEA applique celle avec la **priorité la plus haute** (valeur numérique la plus basse).
- Les admins créés via `pi-manage` (base de données locale) peuvent être ciblés en précisant leur nom dans le champ **User** de la policy, à condition qu'ils soient rattachés à un realm admin.
- Si tu veux éviter la déconnexion automatique mais garder un verrouillage d'écran (plus pratique en administration), laisse `timeout_action` sur `lockscreen`.
